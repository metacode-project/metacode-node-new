# 标签管理模块后端设计（design-server）

## 1. 目标
本设计面向标签管理的后端能力，覆盖标签 CRUD 与列表查询。

## 2. 领域边界
- 模块：`tag-manage`（建议归属 `server/src/modules/tag/`）
- 核心职责：
  - 管理标签实体生命周期（创建、读取、更新、删除）
  - 保证标签名称在业务范围内唯一
  - 提供面向前端页面的查询与变更接口

## 3. 数据模型（逻辑）
标签实体建议包含：
- `id`：唯一标识（字符串或自增 ID）
- `name`：标签名称（必填，唯一）
- `description`：标签描述（可空）
- `createdAt`：创建时间
- `updatedAt`：更新时间

约束建议：
- `name` 唯一索引
- `name` 持久化前做 trim，避免仅空格或伪重复数据

## 4. 接口职责（tRPC）
建议在 `tag` 路由下提供：
- `tag.list`
  - 入参：`keyword?`（可选）
  - 出参：标签列表（可按创建时间倒序）
  - 职责：按关键词（名称）过滤或返回全部
- `tag.create`
  - 入参：`name`、`description?`
  - 出参：新建后的标签数据
  - 职责：校验并创建标签
- `tag.update`
  - 入参：`id`、`name`、`description?`
  - 出参：更新后的标签数据
  - 职责：校验目标存在后更新
- `tag.delete`
  - 入参：`id`
  - 出参：删除结果（成功标记）
  - 职责：校验目标存在后删除

## 5. 数据流
1. 前端通过 `trpc/client` 调用标签接口。
2. RPC 层完成入参校验后进入标签模块服务层。
3. 服务层执行业务规则（唯一性、存在性、字段规范化）。
4. 仓储层读写数据库并返回结果。
5. RPC 层将标准化响应返回前端，前端据此刷新列表与提示状态。

## 6. 鉴权与权限策略
- 默认要求登录态访问。
- 最小权限策略：具备后台管理权限的用户可执行 CRUD。
- 若当前项目尚未细分 RBAC，可先统一校验“已认证后台用户”，后续再细化操作级权限。

## 7. 错误处理策略
- 参数校验失败：返回可读的字段级错误（如名称必填）。
- 业务冲突：名称重复返回冲突错误（可映射为 `CONFLICT`）。
- 资源不存在：更新/删除不存在 ID 返回 `NOT_FOUND`。
- 基础设施异常：统一映射为内部错误并记录日志，返回前端通用失败消息。

## 8. 一致性与可观测性
- 写操作（create/update/delete）建议在单次事务内完成。
- 关键日志字段建议包含：`action`、`tagId`、`operatorId`、`success`、`errorCode`。
- 可补充简单指标：请求成功率、平均响应时间、冲突错误次数。

## 9. 非功能要求
- 列表查询响应应满足后台页面可用性（常规数据量下快速返回）。
- 变更接口需具备幂等容错表现（重复提交时返回清晰错误或成功状态）。
- 错误信息对前端可消费、对日志可排查。
